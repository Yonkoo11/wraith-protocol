Section 01

THE PRIVACY CONTRACT

Precise cryptographic claims. Not marketing promises.

🔒 CRYPTOGRAPHICALLY HIDDEN
Link between deposit and withdrawal ›
Merkle proof in ZK circuit never reveals the leaf index. The prover knows their path; the verifier only sees the root and nullifierHash.
Depositor's Starknet address ›
Not in any public circuit input. msg.sender from the deposit transaction is never passed into the ZK proof's public inputs.
secret, nullifier values ›
Private circuit inputs. They are never transmitted over the network, never stored server-side, never appear in garaga calldata.
Which deposit funded this payment ›
Zero-knowledge property of Groth16. The proof attests "I know a valid preimage for some leaf in this Merkle tree" without revealing which leaf.
👁 PUBLICLY VISIBLE
Depositor address at deposit time ›
Emitted in the on-chain Deposit event. Anyone watching the chain sees your address deposited into the pool. Wraith is link-private, not identity-private.
Deposit amount ›
On-chain, visible to all. Included in the commitment hash but the amount itself is public. Fixed-denomination pools (like Tornado Cash) reduce amount correlation.
Withdrawal recipient address ›
pool.withdraw() recipient is a public input. The server's API wallet address receiving payment is visible on-chain. Only the link to the depositor is hidden.
nullifierHash — required for double-spend prevention ›
Stored on-chain after each withdrawal. Required to prevent the same deposit being spent twice. The nullifierHash is H(nullifier) and does not reveal the original nullifier or depositor.
Payment timestamp (HTTP) ›
Standard HTTP metadata. Timestamps are a known deanonymization vector. The WithdrawalQueue with flushIntervalMs=300000 batches withdrawals to break timing correlation.
⚠ Server logging: X-Payment-Proof header is stripped before handlers. Any upstream proxy logging (nginx access_log, Cloudflare, AWS ALB) must also be audited — a logged X-Payment-Proof header defeats the ZK unlinking entirely.

Section 03

ANONYMITY SET

The most important privacy parameter. Not a feature — a prerequisite.

Which dot funded the withdrawal?
That's the guarantee.

Pool depth risk scale
1–9 HIGH RISK
10–99 MODERATE
100–999 LOW RISK
1000+ MINIMAL

Wraith's privacy guarantee is only as strong as the number of other depositors in the pool. A chain observer sees all deposits. With 3 depositors, they narrow candidates to 3.

Never withdraw immediately after depositing. The default flushIntervalMs=300000 (5 minutes) batches withdrawals to break timing correlation. Setting it to 0 creates a 1-to-1 timestamp link.

Section 04

BUILD WITH WRAITH

SDK for agents and servers. Protocol transport documented below.

agent.ts 
import { CipherPolAgent } from 'cipher-pol-agent';
import { Account, RpcProvider } from 'starknet';

const provider = new RpcProvider({ nodeUrl: RPC_URL });
const account  = new Account(provider, ADDRESS, PRIVATE_KEY);

const agent = new CipherPolAgent({
  adapter:        'privacy-pools',
  poolAddress:   POOL_ADDRESS,
  starknetRpcUrl: RPC_URL,
}, account);

// Deposit once, pay many times
await agent.deposit(1_000_000n); // 0.001 ETH

// Pay any x402-compatible API — no trace to your address
const result = await agent.pay(
  'https://api.example.com/v1/generate',
  { amount: 100n, token: 'ETH' }
);
server.ts 
import express from 'express';
import { cipherPolPaywall }  from 'cipher-pol-server';
import { WithdrawalQueue } from 'cipher-pol-server';

const queue = new WithdrawalQueue({
  account,
  poolAddress:     POOL_ADDRESS,
  vkPath:          './circuits/verification_key.json',
  garagaPath:      './vendor/garaga-v0.15.3',
  flushIntervalMs: 300_000, // 5 min — privacy parameter
});
queue.start();

app.use('/v1/generate', cipherPolPaywall({
  poolAddress:    POOL_ADDRESS,
  requiredAmount: 100n,
  token:          'ETH',
  withdrawalQueue: queue,
}));

// X-Payment-Proof stripped before downstream handlers
// nullifierHash tracked in-memory (use Redis for production)
x402 payment flow (HTTP) 
// 1. Client probes the endpoint
GET /v1/generate

// 2. Server responds with payment requirements
HTTP/1.1 402 Payment Required
X-Payment-Scheme:  cipher-pol-groth16-v1
X-Payment-To:      0x5f71d8ddf...
X-Payment-Amount:  100
X-Payment-Pool:    0x5f71d8ddf...

// 3. Client generates ZK proof (4–6s, client-side)
//    and retries with proof in header
POST /v1/generate

  "zkProof": ["0x1a2b...", "0x3c4d...", ...],
  "nullifierHash": "0x19687381...",
  "publicInputs": { ... }

// 4. Server verifies proof, strips header, routes to handler
HTTP/1.1 200 OK
{ "result": "..." }

// X-Payment-Proof contains ONLY:
//   zkProof (30 felts), nullifierHash, publicInputs

Section 06

HONEST LIMITS

This is a research implementation. The ZK proof is real. The privacy properties are real. The production readiness is not.

Limit 01
ONE-PARTY TRUSTED SETUP
The current .zkey was generated in a single-party Powers of Tau ceremony. If the toxic waste was not destroyed, proofs can be forged and the circuit can be broken without detection. For production: run a multi-party ceremony using Hermez Phase 1 ptau files.
Limit 02
DEPOSITOR ADDRESS IS VISIBLE
When you call pool.deposit(), your Starknet address appears in the Deposit event. Anyone watching the chain knows you participated in this pool. Wraith hides the link between your deposit and the withdrawal — not your participation.
Limit 03
TIMING CORRELATION
Depositing and withdrawing within seconds trivially links them via timestamps. The default flushIntervalMs=300000 (5 minutes) batches withdrawals to break timing correlation. With flushIntervalMs=0, every HTTP request creates a 1-to-1 on-chain timestamp link that defeats the ZK unlinking.
Limit 04
ANONYMITY SET SIZE
If only 3 people have ever deposited into the pool, a chain observer narrows candidates to 3. Privacy scales directly with pool adoption. Do not use Wraith for sensitive payments when pool.size < 50. At pool.size = 1 there is zero anonymity.
