Section 01

The Privacy Contract

Precise cryptographic bounds. Not marketing promises.

🔒 Cryptographically Hidden
Link between deposit and withdrawal ›
Merkle proof in ZK circuit never reveals the leaf index. The prover knows their path; the verifier only sees the root and nullifierHash.
Depositor's Starknet address ›
Not in any public circuit input. msg.sender from the deposit transaction is never passed into the ZK proof's public inputs.
Secret, nullifier values ›
Private circuit inputs. They are never transmitted over the network, never stored server-side, never appear in garaga calldata.
Which deposit funded this payment ›
Zero-knowledge property of Groth16. The proof attests "I know a valid preimage for some leaf in this Merkle tree" without revealing which leaf.
👁 Publicly Visible
Depositor address at deposit time ›
Emitted in the on-chain Deposit event. Anyone watching the chain sees your address deposited into the pool. Cipher Pol is link-private, not identity-private.
Deposit amount ›
On-chain, visible to all. Included in the commitment hash but the amount itself is public. Fixed-denomination pools reduce amount correlation.
Withdrawal recipient address ›
pool.withdraw() recipient is a public input. The server's API wallet address receiving payment is visible on-chain. Only the link to the depositor is hidden.
nullifierHash — for double-spend prevention ›
Stored on-chain after each withdrawal. Required to prevent the same deposit being spent twice. The nullifierHash does not reveal the original nullifier or depositor.
Payment timestamp (HTTP) ›
Standard HTTP metadata. Timestamps are a known deanonymization vector. The WithdrawalQueue with flushIntervalMs=300000 batches withdrawals to break timing correlation.
⚠ Server logging: X-Payment-Proof header is stripped before handlers. Any upstream proxy (nginx, Cloudflare, AWS ALB) must also be audited. A logged X-Payment-Proof header defeats the ZK unlinking entirely.

Section 03

Anonymity Set

The most important privacy parameter. Not a feature. A prerequisite.

Which dot funded the withdrawal?
That's the guarantee.

Pool depth risk scale
1–9 HIGH RISK
10–99 MODERATE
100–999 LOW RISK
1000+ MINIMAL

Cipher Pol's privacy guarantee is only as strong as the number of other depositors in the pool. A chain observer sees all deposits. With 3 depositors, they narrow candidates to 3.

Never withdraw immediately after depositing. The default flushIntervalMs=300000 (5 minutes) batches withdrawals to break timing correlation. Setting it to 0 creates a 1-to-1 timestamp link that defeats the ZK unlinking.

Build with Cipher Pol

SDK for agents and servers. Protocol transport documented below.

agent.ts 
import { CipherPolAgent } from 'cipher-pol-agent';
import { Account, RpcProvider } from 'starknet';

const provider = new RpcProvider({ nodeUrl: RPC_URL });
const account  = new Account(provider, ADDRESS, PRIVATE_KEY);

const agent = new CipherPolAgent({
  adapter:        'privacy-pools',
  poolAddress:   POOL_ADDRESS,
  starknetRpcUrl: RPC_URL,
}, account);

// Deposit once, pay many times
await agent.deposit(1_000_000n); // 0.001 ETH

// Pay any x402-compatible API — no trace to your address
const result = await agent.pay(
  'https://api.example.com/v1/generate',
  { amount: 100n, token: 'ETH' }
);

import express from 'express';
import { cipherPolPaywall }  from 'cipher-pol-server';
import { WithdrawalQueue } from 'cipher-pol-server/withdrawal-queue';

Section 06

Honest Limits

This is a research implementation. The ZK proof is real. Seven documented limits. The production readiness is not.

Limit 01
One-Party Trusted Setup
The current .zkey was generated in a single-party Powers of Tau ceremony. If the toxic waste was not destroyed, proofs can be forged and the circuit broken without detection. Production requires a multi-party ceremony using Hermez Phase 1 ptau files.
Limit 02
Depositor Address is Visible
When you call pool.deposit(), your Starknet address appears in the Deposit event. Anyone watching the chain knows you participated in this pool. Cipher Pol hides the link between your deposit and the withdrawal — not your participation.
Limit 03
Timing Correlation
Depositing and withdrawing within seconds trivially links them via timestamps. The default flushIntervalMs=300000 (5 minutes) batches withdrawals to break timing correlation. With flushIntervalMs=0, every HTTP request creates a 1-to-1 on-chain timestamp link that defeats the ZK unlinking.
Limit 04
Anonymity Set Size
If only 3 people have ever deposited into the pool, a chain observer narrows candidates to 3. Privacy scales directly with pool adoption. Do not use Cipher Pol for sensitive payments when pool.size < 50. At pool.size = 1 there is zero anonymity.
Limit 05
Server Operator Knows Your IP
The ZK proof hides the link between your deposit and payment from the chain. It does not hide your IP address from the server. The server operator has a complete IP-to-payment graph. For strong anonymity, route through Tor or a relay. Unmitigated in v1.
Limit 06
In-Memory Nullifier Set
The double-spend prevention is stored in RAM. A server restart clears it, opening a window where the same note can be spent twice until the next on-chain nullifier sync. Production requires a Redis-backed NullifierSet. The interface is implemented but not tested against live Redis.
Limit 07
No Relay — Operator Has Full Graph
The server submits withdrawals via pool.withdraw(). The operator holds the complete map of nullifierHash → withdrawal tx. Chain observers cannot link deposit to withdrawal — the operator can. A relay network would close this. v1 does not include one.
